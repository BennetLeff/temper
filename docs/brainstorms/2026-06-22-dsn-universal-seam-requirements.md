# DSN/SES Universal Seam for Pipeline Stage Boundaries

## Summary
Formalize SPECCTRA DSN/SES as the canonical intermediate format at every pipeline stage boundary, replacing ad-hoc `.kicad_pcb` workarounds with a universal contract that enables independent stage testing, cross-pipeline composition, and external tool verification.

---

## Problem Frame
Temper has three overlapping pipeline systems (PipelineOrchestrator, RouterV6Pipeline, DeterministicPipeline) that speak different internal data structures. The active strangler-fig effort needs to test that extracted stages produce results identical to their monolith counterparts, but without a common interchange format these parity tests require ad-hoc wrappers that serialize to temporary KiCad PCB files. Meanwhile, 118 ad-hoc scripts each invent their own data-passing convention. A canonical DSN/SES seam makes every pipeline stage boundary speak the same language -- parity tests, stage composition, and external validation all inherit a single contract for free.

---

## Actors
- **A1. Pipeline developer**: Extracts, replaces, or composes individual pipeline stages
- **A2. CI system**: Validates that stage replacement or refactoring does not introduce regressions
- **A3. External tool** (FreeRouting, KiCad DRC): Provides independent routing and design-rule verification
- **A4. Experiment runner**: Runs batch experiments across different pipeline configurations and compares results

---

## Key Flows

### F1. Stage extraction and parity testing
- **Trigger:** Developer extracts a stage from the PipelineOrchestrator monolith
- **Actors:** A1, A2
- **Steps:**
  1. Developer picks a stage boundary (e.g., after placement, after net classification)
  2. Developer records golden DSN output from the monolith at that boundary
  3. Developer implements the extracted stage as a standalone unit
  4. Standalone stage consumes the upstream golden DSN and produces downstream output
  5. CI compares standalone output to monolith golden DSN -- byte-identical means parity
- **Outcome:** Extracted stage proven correct without running the entire monolith
- **Covered by:** R1, R2, R3

### F2. Cross-pipeline stage composition
- **Trigger:** Developer wants to use RouterV6 Stage 2 output as DeterministicPipeline Stage 3 input
- **Actors:** A1
- **Steps:**
  1. RouterV6 Stage 2 writes DSN to a file at its boundary
  2. DeterministicPipeline Stage 3 reads and validates the DSN schema version
  3. Stage 3 processes without any knowledge of RouterV6 internals
  4. Developer verifies the composed pipeline end-to-end against a known golden output
- **Outcome:** Pipeline stages become composable Lego blocks -- any stage can follow any other if the DSN schema matches
- **Covered by:** R4, R5

### F3. External tool verification
- **Trigger:** Any pipeline stage produces DSN output
- **Actors:** A3, A4
- **Steps:**
  1. Pipeline stage writes DSN to a file
  2. FreeRouting CLI imports the DSN and produces SES output
  3. Experiment tracker records routing completion rate, unrouted nets, via counts
  4. KiCad imports the SES output for DRC verification
- **Outcome:** Any stage's output is immediately verifiable by industry-standard tools without custom bridge code
- **Covered by:** R6, R7

## Requirements

### DSN as canonical intermediate format
- **R1.** Every pipeline stage boundary MUST serialize its output to SPECCTRA DSN format before passing to the next stage
- **R2.** DSN serialization MUST be deterministic -- identical inputs produce byte-identical DSN output -- so `git diff` on committed golden files detects regressions with zero false positives
- **R3.** DSN normalization MUST strip non-semantic noise (timestamps, tool version strings, shard-dependent ordering) so diffs compare meaning rather than incidental variation

### Stage boundary contracts
- **R4.** Each stage's DSN schema MUST be versioned with a content hash or semantic version embedded in the DSN file so downstream consumers can detect format changes
- **R5.** A stage consuming DSN input MUST validate the schema version before processing and fail early with a clear diagnostic if the version is incompatible or missing

### External tool interoperability
- **R6.** Pipeline DSN output MUST be directly importable by FreeRouting (open-source SPECCTRA autorouter) without preprocessing -- the same file fed to `freerouting -de <file>.dsn` must produce valid SES output
- **R7.** Pipeline SES (routed) output MUST be importable by KiCad's native SPECCTRA session importer without modification

### Deterministic round-trip parity for pipeline boundaries
- **R8.** For each pipeline stage boundary, a committed golden DSN file MUST exist that captures the monolith's output at that boundary and is regenerated by CI to detect drift
- **R9.** A standalone re-implementation of any stage, given the upstream golden DSN as input, MUST produce output that is byte-identical to the downstream golden DSN (or an approved difference list if the stage intentionally changes behavior)

## Acceptance Examples

- **AE1. Covers R1, R2.** Given a canonical Temper PCB and PipelineOrchestrator running the SEMANTIC phase, the DSN output matches the committed `golden/semantic_output.dsn` byte-for-byte.
- **AE2. Covers R6.** Given a DSN file produced by temper's placement stage, `freerouting -de <file>.dsn` imports it without errors and produces a valid SES file containing at least one routed trace.
- **AE3. Covers R3.** Two DSN files produced on different machines from identical inputs are diff-identical after normalization; only intentional design changes produce diffs.
- **AE4. Covers R4, R5.** A DSN file with a schema version unknown to the consuming stage causes it to fail immediately with a message naming the expected and received versions, rather than producing corrupted downstream output.
- **AE5. Covers R8, R9.** CI runs `temper-verify-boundary --stage=placement` and reports PASS when the standalone placement stage's DSN matches the monolith's golden DSN.

---

## Success Criteria

1. A developer can extract a single pipeline stage, run it standalone, and compare its DSN output to the monolith's golden DSN with `git diff` -- no custom comparison tool needed.
2. FreeRouting can consume placement-stage DSN output and produce routing results without modification or scripted preprocessing.
3. A new pipeline stage implementation can be tested end-to-end without standing up any existing pipeline system -- feed it an upstream golden DSN file and `diff` the output against the downstream golden DSN.
4. The CI pipeline regenerates every golden DSN file on every commit to a golden-input path (board model, design rules, netlist) and fails if any golden file changed unexpectedly.

## Scope Boundaries

- **In scope:**
  - DSN/SES as the canonical interchange format at every pipeline stage boundary
  - Deterministic, diffable serialization with non-semantic noise stripped
  - Schema versioning embedded in DSN output
  - FreeRouting and KiCad import compatibility
  - Golden DSN files committed to the repository for every stage boundary

- **Deferred:**
  - KiCad-specific data that DSN cannot represent (zone fills, custom pad shapes, teardrops, graphical items) -- these require a custom extension format or a parallel sidecar file
  - Full round-trip fidelity (KiCad PCB → DSN → KiCad PCB with zero loss) -- v1 treats DSN as a pipeline internal seam, not archival storage
  - DSN as a real-time streaming protocol -- the seam is file-based

- **Out of scope:**
  - Replacing KiCad's native `.kicad_pcb` format for primary design storage
  - Supporting interchange formats other than DSN/SES (IPC-2581, ODB++, Gerber-based workflows)

---

## Key Decisions

1. **DSN/SES over IPC-2581.** DSN is human-readable, line-diffable, universally supported by autorouters (FreeRouting, SPECCTRA), and the codebase already has a working DSN exporter. IPC-2581 is XML-based, manufacturing-focused, and harder to diff meaningfully.

2. **File-based over in-memory.** File-based seams enable `git diff` as a universal comparison tool, allow direct consumption by external tools without temper-specific libraries, and make golden-file testing trivial. In-memory seams would require custom serialization for every external integration point.

3. **Byte-identical rather than semantic equivalence for parity.** Semantic equivalence (e.g., graph-isomorphic-DSN) is harder to define and implement. Byte-identical output forces deterministic serialization and gives `git diff` zero false positives.

4. **Content hash versioning over sequential version numbers.** Content hashes (e.g., a hash of the DSN schema structure) embed naturally in DSN comment headers and allow consumers to detect drift without a central version registry.

## Dependencies / Assumptions

- **Verified:** The codebase already contains a working DSN exporter (`temper_placer.io.dsn_exporter`) with structure, library, placement, network, and wiring sections. Multiple `.dsn` and `.ses` files exist in `pcb/`, and FreeRouting batch experiment infrastructure already consumes them.
- **Assumes** the existing DSN exporter covers the data fields needed at each pipeline stage boundary -- this must be confirmed by mapping each stage's current internal data structures to DSN sections.
- **Assumes** FreeRouting's DSN parser handles the Temper PCB's layer stack (4-layer with mixed power/ground planes) and design rules without choking on layer-type annotations.
- **Depends** on agreement that DSN/SES is the single canonical interchange format for all pipeline boundaries -- adding parallel formats undermines the universal-seam value proposition.

## Outstanding Questions

### Resolved (pre-planning complete)

- **[R1, R8][Resolved]** The DSN exporter covers structure, library, placement, network, and wiring sections — sufficient for placement-stage and routing-stage boundaries. RouterV6 Stage 2 outputs (occupancy grids, channel capacity maps, bottleneck analysis) are NOT mappable to DSN and use a parallel `Stage2Output.json` sidecar file. Stage boundaries that pass purely geometric data (component positions, netlists, routed traces) use DSN; boundaries that pass analysis artifacts use JSON sidecars. See the full boundary inventory below.
- **[R1][Resolved]** DSN is accepted as lossy for internal analysis data. The universal seam is a family of formats: DSN for geometry/interconnect data, JSON for analysis/derived data. Each stage boundary picks the format appropriate to its output.
- **[R9][Resolved]** Pipeline boundary inventory: PipelineOrchestrator = 8 phase boundaries, RouterV6Pipeline = 5 stage boundaries, DeterministicPipeline = 26 stage boundaries. Total: 39 boundaries. v1 registers the 5 highest-value boundaries first (after SEMANTIC, after TOPOLOGICAL, after placement, after routing, after validation). See the companion doc `docs/brainstorms/2026-06-22-golden-fixture-ladder-requirements.md` for golden fixture generation at each registered boundary.
- **Cross-document integration complete.** This doc pairs with `docs/brainstorms/2026-06-22-golden-fixture-ladder-requirements.md` (the parity-test mechanism DSN enables) and `docs/brainstorms/2026-06-22-unified-stage-protocol-requirements.md` (the typed `StageInput`/`StageOutput` that DSN serializes). ### Deferred to Planning
- **[Affects R2, R3][Technical implementation]** Where does normalization live -- in the DSN serializer, in a post-processing pass, or in a separate normalize-on-read step? The current exporter has some ordering (nets sorted by fanout then span) that may need to become pure alphabetical for determinism.
- **[Affects R4][Technical implementation]** Schema version embedding mechanism -- DSN comment header (`;schema-version: sha256:...`), sidecar `.version` file, or filename convention (`stage3_placement_v2.dsn`). Comment header is the most self-contained.
- **[Affects R5][Technical implementation]** Version validation -- should it be a decorator/wrapper on stage execution, a base-class method, or a standalone validator invoked by each stage?
- **[Affects R3][Technical implementation]** Which fields are "non-semantic noise"? Timestamps and tool versions are obvious; what about per-machine path fragments, optimization-internal seed values, or JAX PRNG state that leaks into output ordering?
