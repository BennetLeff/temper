# Temper Metrics Registry

This document defines all metrics tracked by the GPBM measurement system.

## How to Use

1. **Define new metrics** by adding entries to the appropriate section below.
2. **Reference metrics** in `bd` issue descriptions using the `measurement_targets` YAML block.
3. **Measurements** are collected automatically on `bd close` (via `bd-done`).
4. **Historical data** is stored in `measurements.jsonl`.

### Adding Measurement Targets to Issues

Include a YAML block in your issue description:

```yaml
measurement_targets:
  - metric: fw_test_coverage
    target: ">=80"
  - metric: placer_drc_violations
    target: "==0"
```

## Firmware Metrics

| Metric ID | Description | Target | Source | Command |
|-----------|-------------|--------|--------|---------|
| `fw_test_coverage` | Unit test line coverage | >=80% | pytest/gcov | `cd firmware/test/build && ctest --output-on-failure` |
| `fw_state_machine_tests` | State machine test count | >=37 | Unity | `ctest -N \| grep -c 'test_state'` |
| `fw_integration_tests` | Integration test count | >=30 | Unity | `ctest -N \| grep -c 'test_integration'` |
| `fw_build_size_kb` | Firmware binary size | <=512 | ESP-IDF | `ls -la build/*.bin \| awk '{print $5/1024}'` |
| `fw_stack_usage_pct` | Peak stack usage | <=80% | FreeRTOS | Runtime analysis |
| `fw_heap_fragmentation` | Heap fragmentation ratio | <=20% | FreeRTOS | Runtime analysis |

## Routing Metrics

Routing metrics evaluate the routed PCB quality and validate that good placements produce good routing outcomes. These metrics are generated by `scripts/placement_quality_report.py`.

**Note:** Routing verification requires the `--route` flag and can take 5-30 seconds depending on board complexity. For faster feedback during development, omit `--route` to get placement and DRC metrics only.

| Metric ID | Description | Target | Source | Command |
|-----------|-------------|--------|--------|---------|
| `router_completion_pct` | % of nets successfully routed | ==100% | RoutingVerifier | `python3 scripts/placement_quality_report.py --pcb {pcb} --route --json \| jq .routing_metrics.completion_pct` |
| `router_wirelength_mm` | Total routed trace length (mm) | project-specific | RoutingVerifier | `python3 scripts/placement_quality_report.py --pcb {pcb} --route --json \| jq .routing_metrics.wirelength_mm` |
| `router_via_count` | Total layer-change vias | project-specific | RoutingVerifier | `python3 scripts/placement_quality_report.py --pcb {pcb} --route --json \| jq .routing_metrics.via_count` |
| `router_feasible` | Routing feasibility check | true | RoutingVerifier | `python3 scripts/placement_quality_report.py --pcb {pcb} --route --json \| jq .routing_metrics.routable` |
| `placer_quality_score` | Composite quality score (0-100) | >=80 | quality_score.py | `python3 scripts/placement_quality_report.py --pcb {pcb} --json \| jq .quality_score` |
| `placer_quality_interpretation` | Quality interpretation | "good" or "excellent" | quality_score.py | `python3 scripts/placement_quality_report.py --pcb {pcb} --json \| jq -r .quality_interpretation` |

### Quality Score Interpretation

The `placer_quality_score` combines placement, DRC, and optionally routing metrics into a single 0-100 score:

| Score Range | Interpretation | Meaning |
|-------------|----------------|---------|
| 90-100 | `excellent` | Production-ready, minimal issues |
| 80-89 | `good` | Acceptable, minor improvements possible |
| 60-79 | `ok` | Functional but needs attention |
| 0-59 | `poor` | Significant issues, requires rework |
