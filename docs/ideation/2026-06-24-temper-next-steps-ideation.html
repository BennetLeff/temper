TOP PICK

1. Manifest-Driven Fault Enum Codegen

Firmware safety hardening Confidence 85% Complexity Low
Description
Generate FAULT_LIST X-macro and fault-name string table from firmware/test/traces/manifest.json at build time. FAULT_IGBT_SHORT and FAULT_ADC_STUCK exist in the SIL fault-injection manifest (10 scenarios, ≤2-tick and ≤52-tick latency targets) and in the test runner (test_sm_fault_on_igbt_short, test_sm_fault_on_adc_stuck), but are absent from state_machine.h's FAULT_LIST X-macro. The manifest is already the de facto SSOT for fault existence — the tests depend on it. Extend the existing codegen pipeline (already generates config.h from YAML and transition_table.h from YAML) to read manifest.json and produce the enum, string table, and injection harness entries. Adding a fault to the manifest automatically adds it everywhere.
Basis
direct evidence-firmware-hardening.md:31 — "FAULT_IGBT_SHORT and FAULT_ADC_STUCK absent from FAULT_LIST. grep across firmware/main/ returns zero matches."
Rationale
Two safety-critical fault conditions (IGBT short = catastrophic silicon failure; ADC stuck = loss of temperature sensing) have no production code path. The tests give false confidence — they reference enum values that don't compile. Codegen closes the gap permanently and prevents recurrence: the manifest already owns fault definitions; the X-macro is a stale copy. Let the machine maintain the copy.
Downsides
Requires build-system integration with the codegen step. Adds a dependency from the C build to a Python script reading JSON.

2. SSOT Coordinate Decommissioning — Delete pin.absolute_position()

SSOT consolidation Confidence 90% Complexity Medium
Description
Delete pin.absolute_position() from the Pin type. Force all ~20 call sites through pin_world_position(pin, comp), which correctly applies rotation and side-mirror transforms. Four rotation-blind call sites (fine_pitch_escape.py:135,212, drc_sweep.py:194, via_validation.py:200, unified_router.py:630) compute comp_pos + pin.position without accounting for rotation or bottom-side mounting — producing geometrically wrong world coordinates for rotated and bottom-side components. Add a CI gate (grep for comp.position + pin.position in temper-placer source) to prevent regression.
Basis
direct evidence-ssot-consolidation.md:19-36 — Part 2 of 4-part SSOT plan: "Pin World Position — PARTIAL (4+1 deferred call sites)." The 4 rotation-blind sites compute comp_pos + pin.position without rotation. pin_world_position(pin, comp) applies rotation & side-mirror.
Rationale
This is a geometric correctness issue, not a code-quality cleanup. Every DRC violation report and routing failure diagnosis that passed through via_validation.py:200 or drc_sweep.py:194 may have operated on wrong world coordinates. Fixing the 4 rotation-blind sites is a correctness fix mislabeled as consolidation. Decommissioning the wrong path (not just deprecating it) is the proven SSOT pattern already demonstrated by LayerIndex.
Downsides
~20 call sites need migration. The 4 rotation-blind sites were deferred because they need a pin_world_position_at(pin, comp, position_override) variant — that surface must be designed first.

3. Consolidate 8 A* Implementations to 1 Survivor

SSOT consolidation Confidence 85% Complexity Medium
Description
Delete 7 of 8 A* pathfinding implementations (~8,500 lines across DeterministicAStar, MultiLayerAStar, BidirectionalAStar, astar_core, astar_pathfinding, fast_router, python_astar, maze_router inline). Declare router_v6/astar_pathfinding.py (733 lines, the Router V6 production pathfinder) as the canonical survivor. The code itself says "Always use MultiLayerAStar (has Cython support) — removed old Python-only DeterministicAStar" at sequential_routing.py:1543 — yet DeterministicAStar still exists. SSOT consolidation Part 4 (A* primitives) is stalled — deletion is cheaper than refactoring and achieves the same goal. Run a regression matrix first: benchmark all 8 against temper.kicad_pcb with same iter caps to confirm behavioral equivalence, surfacing which are safe to delete.
Basis
direct evidence-ssot-consolidation.md:61-82 — 8 files, ~9,250 lines, duplicated heuristics, neighbor expansion, and path reconstruction. sequential_routing.py:1543: "Always use MultiLayerAStar."
Rationale
Every new routing feature (via infrastructure, escape routing, multi-layer) currently has 8 places it could be implemented. One survivor eliminates the "which A* do I modify?" question and frees ~8,500 lines of maintenance burden. The regression matrix converts "we think these are dead" into "we know these produce identical output" — the measurement prerequisite the SSOT track needs.
Downsides
The regression matrix must be built and run. Numba fast_router and Cython MultiLayerAStar serve different performance profiles — the survivor may need both code paths under one roof.

4. Hardest-First Net Ordering (Medical Triage)

Placer completion rate Confidence 70% Complexity Low
Description
Route the most-constrained nets first instead of sequentially. The 33% completion wall exists because nets with many degrees of freedom (SPI, PWM) consume scarce routing channels before the 6mm-constrained nets (GATE_H, GATE_L, I_SENSE) have a chance. Triage nets by degrees of routing freedom — how many viable paths exist given clearance constraints — and route the hardest nets first. The netclass data already tells you which nets have HV clearance requirements; no new infrastructure needed. If routing GATE_H first bumps completion from 33% to 50%, that's a 17-point gain from a sort comparator.
Basis
direct exp11_gate_internal_routing/REPORT.md:9-10 — IEC 60335 6mm clearance as root cause for GATE_H, GATE_L, PWM_H, PWM_L, I_SENSE failures.
Rationale
This is the cheapest possible completion-rate improvement — no new data, no new algorithm, just reorder the routing queue. The retry caps and 120s budget in the router are defensive code that exists because the router spends budget on nets that were never going to succeed. Hardest-first ordering removes the need for those caps: the hard nets get first claim on contested channels.
Downsides
Degrees-of-freedom estimation from netclass data is approximate — an HV net between adjacent pads may have more routing freedom than an LV net across the board. The triage score may need refinement.

5. Import-Linter Allowlist Freeze at July 6 Hard-Launch

CI/process hardening Confidence 90% Complexity Low
Description
Freeze the 238-line import-linter allowlist at the July 6 hard-launch. After cutover: no new allowlist entries except via explicit PR approval with a referenced ticket. Auto-escalate any new import that happens to match an existing allowlist regex as a distinct violation — preventing new coupling from hiding behind old waivers. Add a shrinkage-rate metric published as a CI artifact so decomposition velocity is visible. This converts the allowlist from a catch-all into a ratchet that shrinks monotonically, consistent with the CI-gate meta-pattern proven across 9 sprints.
Basis
direct evidence-ci-hardening.md:5-9 — CUTOVER_DATE = datetime.date(2026, 7, 6), soft-launch until then. import-linter-boundary-enforcement-ratchet doc: 238-line allowlist with regex patterns.
Rationale
Without a freeze rule, the 238 waived entries create a shadow baseline that enables new violations to hide behind existing regex patterns. The strangler-fig decomposition stalls because new coupling can always be waived. The freeze is the natural policy — the soft-launch period was for resolving existing violations, not accumulating new waivers.
Downsides
Legitimate new architectural exceptions (e.g., a new package that needs to cross a boundary) must go through explicit approval. The approval process adds friction but the friction is the point.

6. Sidecar-as-Contract Place-Route Feedback Loop

Placer completion rate Confidence 75% Complexity Medium
Description
Formalize placement.channels.json as a required Stage contract (not optional enrichment). After routing, the router MUST produce a bottleneck_report.json sidecar; the placer MUST consume it on the next iteration. The iterative place-route loop skeleton already exists (pipeline/iterative_placer.py, pipeline/iterator.py, pipeline/orchestrator.py with max_iterations=5, routability_threshold=0.85) but has unresolved TODOs at pipeline/feedback.py:129,132. The Stage protocol pattern (declared writes/reads fields) is the exact mechanism to enforce "router writes bottleneck_report, placer reads bottleneck_report." This makes the 7 placement features compound: each feature ships into a system that iterates rather than being a one-shot dead-end optimization.
Basis
direct dsn-universal-seam-eda-pipelines doc lines 79-91 — sidecar pattern documented as optional. evidence-placer-completion.md:128-132 — "Stage-2 channel analysis output is a measurement surface for placement optimization."
Rationale
This is the compound-interest move for the placer axis. The 7 placement features are currently dead-end optimizations — you run once and stop. With the sidecar contract, ghost-pad creates room → router reports remaining bottlenecks → channel-aware scoring consumes that report → next placement is better. The feedback loop multiplies the value of every placement feature shipped into it.
Downsides
The feedback loop adds latency (multiple route-place cycles). The closure test would need to measure convergence rate, not just final completion. If the router's bottleneck report is noisy, the loop may oscillate.

7. Gitignore Root-Anchor + Stale Worktree Cleanup

Repo hygiene & process Confidence 95% Complexity Low
Description
Three concrete hygiene moves: (1) Anchor *.log, *.txt, *.bak to root (/*.log, /*.txt, /*.bak) so they only match root-level artifacts — preventing recursive data loss where a legitimate packages/foo/debug.log would be silently gitignored. (2) Create a worktree deaccessioning policy: for each of the 7 stale worktrees (137-141 commits behind main, last committed June 23), audit whether the code was extracted to main — if yes, remove; if no, extract then remove. (3) Extract useful boards and routing snapshots from the dead 602MB temper-placer/ into tests/fixtures/ as a curated test corpus, then delete the rest. All three are one-time passes with lasting benefit.
Basis
direct evidence-repo-hygiene.md:26-28 — recursive *.log/*.txt/*.bak patterns miss root-anchoring. Lines 36-47 — 7 feature worktrees 137-141 commits behind. Lines 57-68 — dead 602MB temper-placer/ directory.
Rationale
The recursive gitignore patterns are a data-loss risk — a developer adds a debug.log to a package, commits, and git silently ignores it. Root-anchoring forces intent to be explicit. The worktree audit answers the question blocking the placer axis: "what did we already build, and what's still worth extracting?" The fixture extraction gives the closure test a richer corpus so it catches the 33% completion wall before it ships.
Downsides
Root-anchoring means intentional per-package .gitignore files must declare their own ignores — but that's the explicit-intent goal. Fixture extraction from the dead temper-placer/ requires judgment about which artifacts are representative.
